# 晋升评估中心 AI 校准模块

这是一个高保真 AI 产品 Demo，展示“晋升评估中心”中的 AI 校准模块如何帮助评审者复核候选人材料。

Demo 的重点不是让 AI 给出最终晋升判断，而是在评审批次语境中呈现：原 AI 评估可能偏重 KPI、交付周期、绩效评级等显性材料，AI 校准模块会补充整理 mentor、知识沉淀、跨团队协作、长期用户价值等容易遗漏的证据线索，并明确哪些材料已验证、哪些仍需人工确认。

最终晋升决定仍由人工评审委员会根据完整材料作出。AI 在这里只负责发现材料盲区、组织证据和辅助解释。

```bash
npm install
npm run dev
```

## 演示主线

进入页面后，默认选中 Case B。页面结构是：

- 顶部：晋升评估中心系统外壳，显示 AI 校准模块、晋升周期、评审批次和演示数据标识。
- 批次概览：显示 3 个 AI case file、风险分布、待确认事项和材料包状态。
- 左侧：A / B / C case file 复核队列。
- 中间：当前候选人的 AI 校准工作台。
- 右侧：AI 评审材料包与治理记录。

建议按这个顺序讲：

1. 先看批次概览，说明这是一个评审批次中的 AI 校准模块，不是孤立页面。
2. 查看 case file 队列，说明 A / B / C 是三种证据结构，不是候选人排序。
3. 进入默认 Case B，查看原 AI 为什么给出“暂缓晋升”：KPI 92%、交付稳定、绩效 B+，但缺少直接增长突破。
4. 查看材料覆盖盲区，说明校准不是重算结论，而是补齐原评估材料视野。
5. 查看证据 workpaper，每条证据都有来源、片段、可信状态、缺失证据、下一步动作和材料包归属。
6. 查看复核提示，确认低估风险只是补充复核信号。
7. 切换右侧 AI 评审材料包：评审委员版、员工解释版、人工复核版。
8. 切换治理记录，查看输入材料、已验证材料、待确认材料、员工自述处理方式和人工边界。
9. 点击“生成待确认线索”，展示员工补充如何被整理成结构化线索，同时保持待确认状态。

## 三个 Case File

Case A 是“维持推荐”摘要案例。

原 AI 建议推荐晋升，校准后没有提示新增复核风险。它证明这个模块不是默认做正向包装；当显性材料充分、隐性贡献没有明显遗漏时，AI 校准会支持维持原讨论重点。

Case B 是完整闭环案例。

原 AI 建议暂缓晋升，但校准助手发现人才培养、知识沉淀和跨团队协作相关证据集中出现，因此提示可能存在被显性指标低估的风险。该风险只用于补充复核，不改变原建议，也不代表最终晋升结论。

Case C 是“补充评审”摘要案例。

原 AI 建议推荐晋升，但校准助手提示协作反馈、知识沉淀复用和带教稳定性仍有未确认点。它证明校准助手也会提醒评审补看潜在材料缺口。

## 明确不包含

这个 Demo 不包含真实数据接入、账户体系、权限或组织管理、流程性裁决动作、员工排序、训练环节、真实员工隐私数据，也不应被理解为完整企业 HR 平台。

所有候选人、材料和 AI 输出都是 mock data，用来演示产品路径和交互方式。
