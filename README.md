# 晋升评估中心 AI 校准模块

这是一个纯前端的高保真 AI 产品演示，展示“晋升评估中心”中的 AI 校准模块如何帮助评审者复核晋升材料。

产品重点不是让 AI 给出最终晋升判断，而是在评审批次语境中呈现：原 AI 评估可能偏重 KPI、交付周期、绩效评级等显性材料，AI 校准模块会补充整理 mentor、知识沉淀、跨团队协作、长期用户价值等容易遗漏的证据线索，并明确哪些材料已验证、哪些仍需人工确认。

最终晋升决定仍由人工评审委员会根据完整材料作出。AI 在这里只负责发现材料盲区、组织证据和辅助解释。

## 技术栈

- Vite
- React
- TypeScript
- 本地演示数据
- Vitest + Testing Library

项目没有后端服务、数据库、API key、服务端运行时或真实 AI 调用，可以作为静态站点部署。

## 页面结构

当前 Demo 按角色拆成三个清晰视角：

1. `HR 复核工作台`
   - 批次概览
   - A / B / C 复核档案队列
   - 候选人 AI 校准工作台
   - 材料覆盖盲区
   - 证据表
   - 人工复核材料
   - 治理记录

2. `评审委员会材料`
   - 补充评估摘要
   - 已验证证据
   - 待确认线索
   - 建议追问
   - 评审提醒

3. `员工解释与补充`
   - 员工可读解释
   - 系统已识别材料
   - 仍需确认事项
   - 员工补充入口
   - 结构化待确认线索

员工视角不会暴露复核队列、其他候选人、治理记录或评审追问；评审委员会视角不会暴露员工输入框。

## 演示主线

进入页面后，默认打开 `HR 复核工作台`，并默认选中 `档案 B`。

建议按这个顺序讲：

1. 查看批次概览，说明这是晋升评估中心里的 AI 校准模块。
2. 查看 A / B / C 复核档案，说明三者代表不同证据结构，不是候选人排序。
3. 进入默认档案 B，查看原 AI 为什么给出“暂缓晋升”：KPI 92%、交付稳定、绩效 B+，但缺少直接增长突破。
4. 查看材料覆盖盲区，说明校准不是重算结论，而是补齐原评估材料视野。
5. 查看证据表，每条证据都有来源、片段、可信状态、缺失证据、下一步动作和材料包归属。
6. 查看复核提示，确认低估风险只是补充复核信号。
7. 切换到 `评审委员会材料`，查看会议可用摘要、已验证证据、待确认线索和建议追问。
8. 切换到 `员工解释与补充`，查看员工可读解释，并生成待确认线索。

## 三个复核档案

`档案 A` 是“维持推荐”摘要案例。

原 AI 建议推荐晋升，校准后没有提示新增复核风险。它证明这个模块不是默认做正向包装；当显性材料充分、隐性贡献没有明显遗漏时，AI 校准会支持维持原讨论重点。

`档案 B` 是完整闭环案例。

原 AI 建议暂缓晋升，但校准助手发现人才培养、知识沉淀和跨团队协作相关证据集中出现，因此提示可能存在被显性指标低估的风险。该风险只用于补充复核，不改变原建议，也不代表最终晋升结论。

`档案 C` 是“补充评审”摘要案例。

原 AI 建议推荐晋升，但校准助手提示协作反馈、知识沉淀复用和带教稳定性仍有未确认点。它证明校准助手也会提醒评审补看潜在材料缺口。

## GitHub Pages 部署

这是静态前端项目，可以部署到 GitHub Pages。

如果部署地址是仓库子路径，例如：

```text
https://<user>.github.io/<repo>/
```

需要在 `vite.config.ts` 中配置：

```ts
export default defineConfig({
  base: "/<repo>/",
  plugins: [react()],
  test: {
    environment: "jsdom",
    globals: true,
    setupFiles: "./src/test/setup.ts",
  },
});
```

如果部署到自定义域名或根路径，可以保持默认配置。
