# 晋升评估 AI 校准工作台

这是一个一天内完成的高保真 AI 产品 Demo，用来展示“已有 AI 晋升评估结果之上”的校准层。

它不是完整 HR 系统，也不是自动晋升工具。Demo 的重点是让评审者看到：原 AI 评估可能更依赖 KPI、项目交付、绩效评级等显性材料，而校准助手可以把 mentor、知识沉淀、跨团队协作、长期用户价值等容易被忽略的贡献整理出来，形成可审阅、可追问、可人工复核的评审材料。

最终晋升判断仍由人工评审委员会完成。AI 在这里只负责发现盲区、组织证据和辅助解释。

## Demo 入口

```bash
npm install
npm run dev
```

打开 Vite 输出的本地地址。当前开发环境中常用地址是：

```text
http://127.0.0.1:5173
```

## 演示主线

进入页面后，默认选中候选人 B。页面是三栏工作台：

- 左栏：A / B / C 三名候选人的场景切换。
- 中栏：当前候选人的原 AI 评估、校准关注点、证据材料和员工侧说明。
- 右栏：评审委员会可直接使用的摘要、追问问题、未确认事项和人工复核边界。

建议按这个顺序讲：

1. 先看左栏，说明三个候选人不是排名，也不是谁更应该晋升，而是三种不同证据结构。
2. 进入候选人 B，查看原 AI 评估为什么给出“暂缓晋升”：KPI 92%、交付稳定、绩效 B+，但缺少直接增长突破。
3. 对比“原评估看到的内容”和“校准助手提示评审补看的内容”，说明校准不是改分，而是提示评审补看材料。
4. 查看隐性贡献证据。每条证据都有贡献维度、事实摘要、来源、可信状态和下一步动作。
5. 查看复核提示、评审讨论摘要、建议追问和未确认事项。
6. 查看员工可读解释，确认语气是说明性的，不承诺改变结果。
7. 点击“生成待确认线索”，展示员工补充如何被整理成结构化线索，同时保持“待确认”状态。
8. 最后强调人工复核边界：AI 不能确认未进入内部材料的事实，也不能判断贡献是否达到晋升标准。

## 三个候选人的含义

候选人 A 是“维持推荐”摘要案例。

原 AI 推荐晋升，校准后也没有提示新增复核风险。它证明这个工具不是默认帮员工加分；当显性业绩材料已经充分、隐性贡献没有明显遗漏时，校准助手会支持维持原讨论重点。

候选人 B 是完整闭环案例。

原 AI 建议暂缓晋升，但校准助手发现人才培养、知识沉淀和跨团队协作相关证据集中出现，因此提示可能存在被显性指标低估的风险。这个风险只用于补充复核，不改变原建议，也不代表最终晋升结论。

候选人 C 是“补充评审”摘要案例。

原 AI 推荐晋升，但校准助手提示协作反馈、知识沉淀复用和带教稳定性仍有未确认点。它证明校准助手也会提醒评审补看潜在风险，而不是单向做正向包装。

## 明确不包含

这个 Demo 不包含，也不应该暗示包含：

- 真实数据库。
- 登录、权限、组织架构或角色管理。
- 审批流。
- 申诉或仲裁流程。
- 员工排名。
- 自动晋升打分。
- 模型训练。
- 真实员工隐私数据。
- 完整企业 HR 工作台。

所有候选人、材料和 AI 输出都是 mock data，用来演示产品路径和交互方式。

## 交付边界

这个 Demo 的交付重点是可演示的产品闭环，而不是企业级系统完整度。

评审者应该能在不听工程解释的情况下理解三件事：

1. 原 AI 评估可能为什么片面。
2. 校准助手补充了哪些可审阅证据和待确认线索。
3. 哪些内容必须继续由人工评审确认。

只要后续文档或演示继续保持这条边界，就不要把它扩展成通用 HR dashboard、申诉系统或自动评分器。
