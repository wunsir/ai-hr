# Product Spec: 晋升评估中心 - AI 校准模块

## Problem / User Need

企业已上线 AI 晋升评估系统，评估效率提升，但系统更容易依赖显性业绩数据，例如 KPI、项目交付、绩效评级等。这会导致 mentorship、知识分享、跨团队支持、组织影响力、长期用户价值等隐性贡献被低估，评估结果显得片面。

同时，部分高潜员工会因为评估过程不透明、结论机械、贡献未被看见而降低对晋升通道的信任。这形成了“效率提升”与“人才感知”之间的矛盾。

本 MVP 要解决的不是“让 AI 自动决定谁晋升”，而是在已有晋升评估系统中增加一个 AI 校准模块。该模块嵌入“晋升评估中心”的评审批次、候选人复核队列、评审材料包和治理记录中，帮助评审识别可能被显性指标低估的贡献，并给员工提供有限但清晰的反馈与补充入口。

## Target User / Scenario

主要用户是 HR、HRBP、晋升评审委员会或业务管理者。他们需要在一个晋升评审批次中查看候选人复核队列、检查原 AI 评估结论、复核 AI 校准助手生成的证据材料，并判断哪些事项需要人工继续确认。

次要用户是被评估员工。员工需要理解系统看见了哪些贡献、哪些材料被采纳、哪些内容仍需确认，并能对遗漏贡献提交补充说明。

核心场景：评审者进入“晋升评估中心”的某个评审批次。该批次包含 3 名候选人的 AI 复核 case file。候选人 A 显性业绩强、隐性贡献一般，原系统推荐晋升，校准后维持推荐；候选人 B 显性业绩中等、隐性贡献强，原系统暂缓，校准助手提示被低估风险；候选人 C 显性业绩强，但协作、带教、知识沉淀或组织影响存在风险，校准助手提示需补充评审。

其中候选人 B 是完整深度案例，用来展示从原 AI 评估、隐性贡献证据、低估风险、评审委员会摘要、员工解释、员工补充、待确认线索到人工复核边界的完整闭环。候选人 A 和 C 用来证明产品不是单向“帮员工加分”，而是在不同候选人类型中帮助评审看到更完整的证据结构。

Demo 需要显得像一个企业 HR 系统中的真实 AI 模块，而不是一张单页信息展示。系统感来自评审批次总览、候选人复核队列、AI 校准工作台、评审材料包和治理记录，但这些均为 mock data 驱动的高保真演示，不扩展为完整 HR 平台。

## Desired Outcome

MVP 需要证明：在一个企业晋升评估系统中，AI 可以从企业已有材料和员工补充说明中提取隐性贡献证据，提示原评估结论可能存在的信息盲区或补充评审点，并生成可审阅、可解释、可人工复核的评审材料包。

成功的产品体验应让评审者更完整地理解候选人贡献，也让员工感到系统不是单纯依靠冷冰冰的显性指标做判断。

为了让 Demo 具备高保真产品感，MVP 还应展示 AI 如何为评审者生成“建议追问问题”和“未确认事项”。这能把产品从静态总结工具提升为评审决策辅助工具，同时保持 AI 不做最终裁决的边界。

为了让 Demo 足够完整和厚重，MVP 应呈现为“HR 晋升评估中心中的 AI 校准模块”：用户先看到评审批次与复核队列，再进入候选人 B 的 AI 校准工作台，最后查看面向评审、员工和人工复核的材料包及治理记录。

## Core User Journey

1. 评审者进入晋升评估中心，看到当前晋升周期、评审批次、候选人数量、AI 复核状态和演示数据说明。
2. 评审者查看候选人复核队列，快速理解 A、B、C 三个 case file 的原 AI 评估结论、校准状态、未确认事项和下一步建议动作。
3. 评审者进入候选人 B 的 AI 校准工作台，查看原 AI 晋升评估结果，包括显性绩效摘要和初步建议。
4. 评审者启动或查看已完成的 AI 校准分析。
5. 系统展示材料覆盖盲区，并从内部材料中提取隐性贡献证据，按贡献维度归类。
6. 系统标注每条证据的来源和可信状态，例如已验证、待确认、证据不足。
7. 系统提示候选人 B 存在“被显性指标低估”的风险。
8. 系统生成评审材料包，包含评审委员版、员工解释版和人工复核版。
9. 评审者查看面向评审委员会的补充评估摘要，并获得建议追问问题和未确认事项。
10. 员工查看可读解释摘要，了解哪些贡献被识别、哪些仍需确认。
11. 员工可以提交一段贡献补充说明。
12. AI 展示员工原始补充说明与结构化贡献线索的前后对比，并提示需要补充的证据来源。
13. 治理记录展示材料来源、AI 输出时间、员工自述状态和人工确认边界。
14. 评审者看到该补充线索，但最终是否采纳由人工确认。

## Must-Have Scope

- 支持 3 名候选人案例切换，展示不同类型的评估结果和校准关注点。
- 展示一个轻量但完整的晋升评估中心外壳，包括晋升周期、评审批次、候选人复核队列、AI 复核状态和演示数据说明。
- 以 case file 形式呈现候选人 A/B/C，而不是简单信息卡片；每个 case file 至少展示原 AI 建议、校准状态、证据覆盖、未确认事项数量和下一步评审动作。
- 候选人 A：显性业绩强，隐性贡献一般，原系统推荐晋升，校准后维持推荐。
- 候选人 B：显性业绩中等，隐性贡献强，原系统暂缓，校准助手提示被低估风险。
- 候选人 C：显性业绩强，但协作、带教、知识沉淀或组织影响存在风险，校准助手提示需补充评审。
- 候选人 B 必须展示完整深度闭环：原 AI 评估、隐性贡献证据、低估风险、评审委员会摘要、员工解释、员工补充、待确认线索、人工复核边界。
- 候选人 B 的深度案例应呈现为 AI 校准工作台，体现材料覆盖、证据抽取、低估风险、材料包生成和人工复核边界的过程。
- 展示每名候选人的原 AI 评估结果，体现原系统偏重显性绩效数据。
- 展示原评估结论与校准后关注点的对比。
- 展示材料覆盖盲区，例如原系统主要覆盖 KPI、交付、绩效评级，但未充分覆盖 mentor 记录、知识分享、跨团队支持和长期用户价值。
- 展示候选人的内部材料样例，包括 360 反馈、项目复盘、知识分享记录、mentor 记录或跨团队协作评价。
- AI 输出隐性贡献提取结果，至少覆盖 mentorship、知识分享、跨团队协作或组织影响力中的若干项。
- 将隐性贡献映射到晋升相关维度，例如用户价值、协作影响、创新突破、知识沉淀、人才培养、长期组织贡献。
- 标注每条证据的来源和可信状态。
- 输出“被低估风险”提示，表达为辅助复核信号，而不是最终晋升判断。
- 生成一份评审委员会可读的补充评估摘要。
- 生成一份员工可读的解释摘要。
- 生成一个“AI 评审材料包”，至少包含评审委员版、员工解释版和人工复核版三种视角。
- 展示轻量治理记录，包括材料来源、AI 输出时间、哪些内容为已验证材料、哪些内容为员工自述、哪些内容需要人工确认。
- 提供员工“贡献补充”入口，并将员工自述标记为待确认线索。
- 展示员工补充说明被 AI 改写为结构化贡献线索的前后对比。
- 展示缺失证据提示，例如需要主管确认、需要项目复盘来源、需要协作方反馈等。
- 为候选人 B 和 C 生成建议评审追问问题，帮助评审者知道下一步应向谁确认什么。
- 明确展示 AI 不能确认的事项，避免把推断包装成事实。
- 明确最终评估结果由人工评审决定，AI 只负责发现盲区、组织证据和辅助解释。

## Nice-to-Have Scope

- 展示不同证据状态的筛选或分组。
- 展示简短的迭代记录，说明如何从“自动评分”调整为“辅助校准”。
- 展示候选人 A、B、C 的横向对比摘要，帮助评审者理解校准助手如何处理不同类型的候选人。
- 展示员工补充说明进入评审侧后的状态变化，例如从“待确认”到“已确认”或“证据不足”。
- 展示批次级别的复核进度，例如“3 个 case，1 个低估风险，1 个补充评审，1 个维持推荐”。
- 展示材料包视角切换后的简短差异，例如评审委员版更强调证据和追问，员工解释版更强调可理解反馈，人工复核版更强调不能确认的事项。

## Non-Goals

- 不做完整 HR 系统。
- 不做真实登录、注册、权限和组织架构管理。
- 不接入真实企业数据库。
- 不处理真实员工隐私数据。
- 不训练模型。
- 不做完整晋升流程、审批流或申诉流程。
- 不做真实审计系统或合规留痕系统。
- 不做员工排名。
- 不替代晋升委员会做最终判断。
- 不承诺算法绝对公平。
- 不证明候选人一定应该晋升。

## Anti-Requirements: What This Should Not Become

- 不应变成“AI 自动晋升打分器”。
- 不应变成“员工申诉仲裁系统”。
- 不应变成“价值观评分器”。
- 不应因为增加 HR 系统外壳而变成完整 HR 平台、绩效管理系统或审批系统。
- 不应出现“通过”“驳回”“提交审批”“申诉成功”等暗示真实流程闭环的动作。
- 不应变成只会生成漂亮总结的文案工具。
- 不应把员工自述直接当作已验证事实。
- 不应暗示隐性贡献越多就必然应该晋升。
- 不应读取高度敏感或不合适的数据来源，例如私人聊天内容。
- 不应使用“公平裁判”“消除偏见”等过度承诺表达。
- 不应在显性名称中绑定某一家公司，但可以在贡献维度和业务案例中体现大型互联网企业的协作、用户价值和组织贡献语境。

## Product Acceptance Criteria

- 用户能在一个完整流程中理解：原 AI 评估为何可能片面，以及校准助手补充了哪些贡献证据。
- 用户能感知这是企业晋升评估中心中的一个 AI 校准模块，而不是孤立的单页展示。
- 用户能先看到评审批次和候选人复核队列，再进入具体候选人的 AI 校准工作台。
- 用户能在 3 名候选人之间切换，并理解 A、B、C 分别代表维持推荐、被低估风险、需补充评审三种典型场景。
- 用户能理解每个候选人 case file 的原 AI 建议、校准状态、证据覆盖、未确认事项和下一步评审动作。
- 用户能清楚区分显性绩效数据、内部已验证证据、员工待确认补充三类信息。
- 用户能看到原系统的材料覆盖盲区，而不只是看到校准后的摘要。
- 用户能看到至少一条隐性贡献如何从原始材料转化为可评审证据。
- 用户能看到原评估结论与校准后关注点的对比。
- 用户能看到“被低估风险”是复核提示，而不是自动改分或自动晋升结论。
- 评审者能获得一份可直接用于讨论的补充评估摘要。
- 评审者能查看 AI 评审材料包，并区分评审委员版、员工解释版和人工复核版的用途。
- 评审者能看到建议追问问题和未确认事项，并理解哪些内容需要人工继续复核。
- 评审者能看到轻量治理记录，理解材料来源、AI 输出和员工自述的可信边界。
- 员工能获得一份可理解、不对抗、不过度承诺的解释摘要。
- 员工提交补充说明后，系统能展示从原始自述到结构化贡献线索的变化，但不会直接采纳为事实，而是将其标记为待确认线索。
- 系统能提示缺失证据，例如需要主管确认、项目复盘来源或协作方反馈。
- 产品全程明确 AI 的边界：发现盲区、组织证据、辅助解释；最终决策由人工完成。
- MVP 能同时回应两个任务矛盾：降低显性数据造成的评估片面性，并通过解释与有限参与感缓解员工对机械评估的不信任。
- Demo 不依赖真实数据库、真实员工数据、复杂权限或完整 HR 流程，也能表达完整产品价值。

## Key Inputs / User-visible Outputs

关键输入：

- 候选人的显性绩效摘要。
- 原 AI 晋升评估结果。
- 当前晋升周期和评审批次信息。
- 候选人复核队列状态。
- 企业内部材料样例，例如 360 反馈、项目复盘、知识库记录、mentor 记录、跨团队协作评价。
- 员工补充说明。

用户可见输出：

- 原评估结果摘要。
- 晋升评估中心总览。
- 评审批次摘要。
- 候选人 case file 复核队列。
- 3 名候选人的校准状态摘要。
- 材料覆盖盲区。
- 原评估结论与校准后关注点对比。
- 隐性贡献证据列表。
- 贡献维度映射。
- 证据来源与可信状态。
- 被低估风险提示。
- 建议评审追问问题。
- AI 未确认事项和缺失证据提示。
- 评审委员会补充评估摘要。
- AI 评审材料包，包括评审委员版、员工解释版和人工复核版。
- 轻量治理记录，包括材料来源、AI 输出时间、员工自述状态和人工确认边界。
- 员工可读解释摘要。
- 员工补充说明的原始内容、结构化结果和待确认提示。

## Open Questions

暂无阻塞性开放问题。

已确认的范围决策：

- 当前范围采用 3 名候选人切换，其中候选人 B 做完整深度闭环。
- 当前范围采用“HR 晋升评估中心中的 AI 校准模块”形态，允许增加系统外壳、批次、队列、材料包和治理记录来增强完整度。
- 当前范围采用隐性的大型互联网企业语境，不在产品名称中显性绑定某一家公司。
- 当前范围展示员工补充说明的输入、结构化结果和待确认状态。

## Handoff to UI Shape

后续 UI shaping 应以本规格为边界，重点表达一个高保真的 HR 晋升评估中心中的 AI 校准模块：评审批次总览、候选人 case file 复核队列、候选人 B 的 AI 校准工作台、原评估结论与校准后关注点对比、隐性贡献证据、低估风险提示、AI 评审材料包、员工解释与贡献补充、轻量治理记录。

UI 可以表现为 HR 系统中的完整模块，但不应把产品表现成真实完整 HR 平台、自动评分系统、审批系统或申诉系统。视觉和文案应强调克制、可信、可解释、可复核。员工侧表达应中性，避免对抗性语言。

## Handoff to Engineering Planning

后续工程 planning 应将本规格作为产品范围来源，保留 must-have、non-goals 和 anti-requirements。可以增强 HR 系统外壳和演示完整度，但不主动扩大为真实数据接入、权限系统、审批流、申诉系统、审计系统或模型训练。

实现层面可以使用 mock data 和 mock AI output 来证明产品价值。工程计划应优先保证主流程完整、输出可解释、边界清晰，以及足够的企业系统感，而不是追求真实系统完整度。

工程实现应优先追求高保真、可演示、可交互的 AI 产品 Demo，而不是最小静态页面；可以使用 mock data 和 mock AI output 完成完整体验闭环。
