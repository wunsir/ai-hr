# UI Shape: 晋升评估中心中的 AI 校准模块

## 1. Product Surface

这是一个一天内可完成的高保真 AI 产品 Demo，形态应从“单页校准工作台”升级为“HR 晋升评估中心中的 AI 校准模块”。用户不只是看到一页候选人材料，而是进入一个企业晋升评审系统的某个评审批次，在批次上下文中查看候选人 case file 队列、进入候选人 B 的 AI 校准工作台、生成 AI 评审材料包，并查看轻量治理记录。

它仍然不是完整 HR 系统。系统外壳只用于建立真实产品语境，不能扩展为登录权限、组织架构、审批流、申诉流、员工排名、真实审计系统、数据库接入或模型训练。

主要用户是 HR、HRBP、晋升评审委员会和业务管理者。次要用户是被评估员工，员工只通过“员工解释版材料包”和“贡献补充入口”获得有限参与感，不进入完整评审中心。

产品要证明：AI 校准模块不是替代评审结论，而是在现有晋升评估中心中发现材料盲区、组织证据、生成可审阅材料包，并把哪些内容已验证、哪些内容待确认、哪些内容 AI 不能确认清楚呈现出来。

## 2. Design Direction

界面类型：product register。设计服务于高风险评审任务，应有企业系统感、材料可信感和证据可追溯感。它要比普通 HR dashboard 更像“评审材料控制台”，比 AI 总结页更像“证据与治理工作流”，比审批系统更克制。

物理场景：HRBP 在评审会前 30 分钟，用 14 到 16 英寸笔记本准备某个技术序列晋升批次；她需要快速判断哪些候选人的材料可直接讨论，哪些需要补充确认，哪些 AI 输出不能作为事实。

视觉策略：Restrained with a precise accent。浅色工作台底色，冷静的灰蓝中性色，少量状态色用于 case 类型、证据可信状态、材料包视角和治理记录。不要使用大面积渐变、发光、玻璃态、装饰插画或仪表盘式大数字。

参考气质：

- Linear 的任务队列密度和状态识别。
- Stripe Dashboard 的可信企业系统感和边界提示。
- Notion database 的材料组织能力，但要更像评审系统，不像个人知识库。
- Vanta 或 WorkOS 的合规材料感，用于治理记录，但不要做真实审计产品。

要在竞品中脱颖而出的关键不是炫酷视觉，而是“证据链体验”足够明确：

- 批次层能看出这不是孤立页面。
- 队列层能看出每个 case file 的复核状态和下一步动作。
- 工作台层能看出原评估盲区、证据抽取、复核提示和材料包生成。
- 治理层能看出 AI 输出、员工自述和人工确认之间的边界。

## 3. Information Architecture

推荐采用“系统外壳 + 批次总览 + case file 队列 + AI 校准工作台 + 材料包侧栏”的复合工作台，而不是简单三列页面。

### 3.1 System Shell

最外层是轻量 HR 系统外壳，用来建立产品语境。

顶部栏展示：

- 产品域：晋升评估中心。
- 当前模块：AI 校准。
- 晋升周期：2026 H1 技术序列。
- 评审批次：P6 到 P7 晋升复核批次。
- 数据标识：演示数据。

顶部栏不展示真实账号、权限切换、组织树、审批入口、设置中心或全局 HR 导航。可以出现不可点击的模块标签，例如“评审批次”“AI 校准”“材料包”，但不要让 Demo 变成完整平台。

### 3.2 Batch Overview Band

顶部下方放一个批次总览带，作为第一屏的系统感来源。它不应做成传统 KPI dashboard，而应像“批次控制条”。

批次总览展示：

- 当前批次：技术序列 2026 H1 复核批次。
- case file 数量：3 个候选人 case。
- AI 复核状态：1 个低估风险、1 个补充评审、1 个维持推荐。
- 待确认事项：例如 6 项待人工确认。
- 材料包状态：候选人 B 已生成评审材料包。

这些数据只是当前 Demo 的 mock 状态，不要扩展成统计大屏。避免大号数字和“全局人才洞察”这类泛化表达。

### 3.3 Case File Queue

批次总览下方左侧或上方横向区域是候选人 case file 队列。它替代原先的“候选人卡片切换”。

每个 case file 必须显示：

- 候选人编号：A/B/C。
- 职能背景：例如商家平台、交易基础设施等。
- 原 AI 建议：推荐晋升或暂缓晋升。
- 校准状态：维持推荐、低估风险、补充评审。
- 证据覆盖：显性绩效、隐性贡献、员工补充三类材料的覆盖状态。
- 未确认事项数量。
- 下一步评审动作。

case file 的视觉应像评审队列中的工作项，而不是营销卡片。它可以采用紧凑表格行、dossier row 或 queue card。不要出现头像墙、排名、总分、通过率、晋升概率。

### 3.4 Candidate Workspace

候选人 B 默认选中，进入中间主工作区。这里是 AI 校准工作台，不是候选人详情页。

候选人 B 工作区应分成四个层次：

1. Case Header：候选人身份、原 AI 建议、校准状态、复核优先级、材料包状态。
2. Calibration Path：原 AI 评估 → 材料覆盖盲区 → 隐性贡献证据 → 复核提示 → AI 材料包。
3. Evidence Workpaper：证据列表、来源、状态、下一步动作、缺失证据。
4. Employee Loop：员工解释、员工补充、结构化待确认线索。

候选人 A 和 C 保持摘要案例，但也要在同一工作区框架中展示，证明它们也是 case file，而不是独立卡片。

### 3.5 Material Package And Governance Side Rail

右侧侧栏不再只是“评审摘要”，而是“AI 材料包 + 治理记录”。

侧栏分为两个切换视角：

- AI 评审材料包。
- 治理记录。

AI 评审材料包内有三个子视角：

- 评审委员版：用于委员会讨论，强调已验证证据、待确认线索、建议追问。
- 员工解释版：用于员工理解，强调材料来源、仍需确认事项和人工评审边界。
- 人工复核版：用于后续确认，强调不能确认事项、缺失证据、建议找谁确认。

治理记录展示轻量 trace，不做真实审计系统：

- 材料来源清单。
- AI 输出时间。
- 员工自述状态。
- 已验证材料数量。
- 待确认线索数量。
- 人工确认边界。

## 4. Default Screen Composition

默认打开“晋升评估中心 > AI 校准”，选中候选人 B。

桌面端第一屏建议结构：

- 顶部：系统外壳栏。
- 第二层：批次总览带。
- 左侧：case file 队列。
- 中间：候选人 B AI 校准工作台。
- 右侧：AI 材料包与治理记录侧栏。

视觉重心从上到下应是：

1. 这是哪个评审批次。
2. 当前批次有哪些 case file。
3. 候选人 B 为什么需要重点复核。
4. AI 从哪些材料中抽取了哪些证据。
5. 这些材料如何进入评审、员工解释和人工复核。

不要把首屏做成“标题 + 三个卡片 + 内容列表”。竞品区分度来自批次语境、队列语义、证据链和材料包，而不是更多卡片。

## 5. Batch Overview

批次总览是系统感的核心，但必须克制。

建议模块名称：评审批次概览。

内容结构：

- 批次信息：2026 H1 技术序列，P6 到 P7 晋升复核。
- 复核范围：3 个 AI case file。
- 当前状态：AI 校准分析已完成。
- 风险分布：低估风险 1，补充评审 1，维持推荐 1。
- 待确认事项：6 项。
- 材料包：候选人 B 已生成，A/C 为摘要材料。

批次总览应使用小型状态 chips 和紧凑说明，不使用仪表盘图表。不要出现“晋升率”“通过率”“人才排名”“高潜排行”。

建议一句批次说明：

> 本批次仅展示 AI 校准材料结构，不代表最终晋升结论。所有待确认线索需由人工评审继续复核。

## 6. Case File Queue

case file 队列是 A/B/C 切换的升级版。它要让用户一眼看出三种典型场景，同时像真实评审队列。

### 6.1 Case File Fields

每个 case file 行展示：

- Case ID：Case A / Case B / Case C。
- 候选人：候选人 A/B/C。
- 职能背景：例如商家平台 · 资深后端工程师。
- 原 AI 建议。
- 校准状态。
- 证据覆盖：显性绩效、内部材料、员工补充。
- 待确认数。
- 下一步动作。

### 6.2 Candidate A

场景：维持推荐。

展示重点：

- 原 AI 建议：推荐晋升。
- 校准状态：维持推荐。
- 证据覆盖：显性绩效充分，隐性贡献未发现明显遗漏。
- 待确认事项：低或 0 到 1。
- 下一步动作：维持原讨论重点。

文案：

> 校准助手未提示新增复核风险。当前材料结构支持围绕业务结果和稳定交付继续讨论。

### 6.3 Candidate B

场景：低估风险。

展示重点：

- 原 AI 建议：暂缓晋升。
- 校准状态：低估风险。
- 证据覆盖：显性绩效中等，隐性贡献证据集中，员工补充有待确认线索。
- 待确认事项：中等，例如 3 项。
- 下一步动作：生成并审阅 AI 材料包，向主管和协作方确认。

文案：

> 多条内部材料指向人才培养、知识沉淀和跨团队协作贡献。建议补充复核，不改变原 AI 建议。

### 6.4 Candidate C

场景：补充评审。

展示重点：

- 原 AI 建议：推荐晋升。
- 校准状态：补充评审。
- 证据覆盖：显性业绩强，但协作、知识沉淀或带教材料存在未确认风险。
- 待确认事项：中等，例如 2 到 3 项。
- 下一步动作：确认协作反馈、知识复用和带教稳定性。

文案：

> 校准助手提示补充确认组织影响材料。该提示不否定原推荐，也不是负面裁定。

### 6.5 Queue Interaction

点击 case file 后：

- 中间工作区切换到对应候选人。
- 右侧材料包和治理记录同步切换。
- 批次总览保持不变。

候选人 B 默认选中。A/C 是摘要视图，不进入完整员工补充闭环。

## 7. Candidate B AI Calibration Workspace

候选人 B 是完整深度案例。它应像一个 AI 校准工作台，而不是一张详情页。

### 7.1 Case Header

顶部展示：

- 候选人 B。
- 商家平台 · 资深后端工程师。
- 原 AI 建议：暂缓晋升。
- 校准状态：低估风险。
- 复核优先级：中。
- 材料包状态：已生成。

Case Header 下放一条边界提示：

> 该状态用于提示评审补充复核，不改变原 AI 建议，也不代表最终晋升结论。

### 7.2 Calibration Path

用一条横向或纵向链路展示完整闭环：

1. 原 AI 评估。
2. 材料覆盖盲区。
3. 隐性贡献证据。
4. 被低估风险提示。
5. AI 评审材料包。
6. 员工解释。
7. 员工补充。
8. 待确认线索。
9. 人工复核边界。

链路不是进度条，不表示流程已审批完成。它是材料结构导航，用于帮助评审理解证据如何从原始材料进入材料包。

### 7.3 Original AI Assessment

展示原 AI 评估结果，重点表达原系统偏重显性绩效。

内容：

- 原建议：暂缓晋升。
- 显性绩效：项目交付稳定、核心 KPI 达成 92%、绩效评级 B+、缺少直接业务增长突破。
- 原评估依据：KPI、交付周期、绩效评级、项目负责人反馈。
- 原评估限制：材料覆盖范围有限，不等于原评估错误。

### 7.4 Material Coverage Gap

这是新版 PRD 的关键新增点。应放在原 AI 评估之后、证据列表之前。

展示为“材料覆盖矩阵”或“coverage ledger”，而不是普通总结。

三组材料：

- 已覆盖显性材料：KPI、交付周期、绩效评级、项目负责人反馈。
- 原系统未充分覆盖：mentor 记录、知识分享、跨团队支持、长期用户价值。
- 员工补充材料：客服工单归因模板线索，当前待确认。

每项材料标注：

- 材料类型。
- 来源。
- 覆盖状态：已覆盖、校准补充、待确认。
- 是否可进入评审讨论。

文案：

> 校准助手不是重算结论，而是补齐原评估材料视野，帮助评审看到显性绩效以外的证据结构。

### 7.5 Original Vs Calibration Comparison

双栏对比仍保留，但名称要更像企业系统：

左侧：原 AI 评估依据。

- 显性绩效中等。
- 缺少直接增长指标突破。
- 晋升建议为暂缓。

右侧：AI 校准提示补看的材料。

- 已验证材料指向人才培养贡献。
- 知识库和复盘记录显示知识沉淀影响多个团队。
- 跨团队协作反馈显示其解决平台接入阻塞。
- 员工补充形成一条待确认长期用户价值线索。

右侧不能叫“修正结果”“改分”“重新判定”。

### 7.6 Evidence Workpaper

证据区应成为页面的专业核心。它应像评审 workpaper，而不是普通卡片列表。

每条证据必须展示：

- 贡献维度。
- 可评审事实。
- 原始来源。
- 来源片段。
- 可信状态。
- 缺失证据。
- 下一步确认动作。
- 是否进入材料包。

建议字段：

| 维度 | 可评审事实 | 来源 | 可信状态 | 缺失证据 | 下一步动作 | 材料包 |
| --- | --- | --- | --- | --- | --- | --- |
| 人才培养 | 连续两个季度担任 2 名 P6 工程师 mentor，其中 1 人已独立负责灰度方案设计 | mentor 记录、主管反馈 | 已验证 | 无关键缺口 | 可纳入评审讨论 | 评审委员版 |
| 知识沉淀 | 商家侧异常归因手册被 4 个团队在项目复盘中引用 | 知识库、项目复盘 | 已验证 | 持续引用频率可补充 | 可纳入评审讨论 | 评审委员版 |
| 协作影响 | 支付链路改造中协调多团队，可能缩短接入排期约 2 周 | 跨团队协作评价 | 待确认 | 影响范围和排期口径 | 向项目负责人确认 | 人工复核版 |
| 用户价值 | 员工补充称其推动客服工单归因模板，可能降低重复定位成本 | 员工补充说明 | 待确认 | 使用范围、效率幅度、正式材料来源 | 需主管或客服团队确认 | 员工解释版、人工复核版 |

证据状态：

- 已验证：可进入评审讨论，但不等于自动支持晋升。
- 待确认：可作为追问线索，不能作为事实采纳。
- 证据不足：只能提示缺口，不应包装成贡献事实。

### 7.7 Underestimation Review Prompt

标题：

> 复核提示：显性指标可能未覆盖部分组织贡献

正文：

> 校准助手发现多条内部材料指向人才培养、知识沉淀和跨团队协作贡献。由于原评估主要依据 KPI、交付和绩效评级，候选人 B 可能存在被显性指标低估的风险。该提示仅用于评审补充复核，不改变原 AI 建议，也不代表晋升结论。

必须同时展示：

- 触发原因。
- 不能确认事项。
- 建议追问对象。
- 人工边界。

不要展示风险分、公平指数、晋升概率、校准后分数。

## 8. AI Review Material Package

AI 评审材料包是新版界面的差异化亮点。它要让产品从“证据页”变成“评审可用模块”。

位置：右侧侧栏或主工作区中的固定材料包面板。

材料包有三个视角，使用 tabs 或 segmented control：

### 8.1 评审委员版

用途：评审会讨论。

内容：

- 补充评估摘要。
- 已验证证据。
- 待确认线索。
- 建议追问问题。
- 评审提醒。

样例结构：

- 摘要：候选人 B 的显性绩效中等，但多条内部材料显示其在人才培养、知识沉淀和跨团队协作中有可讨论贡献。
- 已验证证据：mentor 支持、异常归因手册、故障定位清单。
- 待确认线索：支付链路排期影响、客服工单模板使用范围。
- 建议追问：向直属主管、支付链路项目负责人、知识库负责人确认。
- 边界：建议补充复核，不替代委员会判断。

### 8.2 员工解释版

用途：员工可理解反馈。

内容：

- 系统看到了哪些材料。
- 哪些材料已来自内部记录。
- 哪些内容仍需确认。
- 员工补充如何被处理。
- 最终结论由人工评审决定。

语气必须中性，不要说“你被低估了”“系统认可你的贡献”“结果会改变”。

样例：

> 本次校准助手识别到你在 mentor 支持、知识沉淀和跨团队协作方面的若干材料。其中，mentor 记录和知识库引用已来自内部材料；支付链路协作影响和你补充的用户价值线索仍需进一步确认。评审委员会会结合已验证材料、待确认线索和原评估结果进行综合讨论，最终结论由人工评审决定。

### 8.3 人工复核版

用途：指导 HRBP 或评审委员下一步确认。

内容：

- AI 不能确认的事项。
- 缺失证据。
- 建议确认对象。
- 需要补充的材料来源。
- 员工自述状态。

样例：

- 不能确认：支付链路排期是否确实缩短约 2 周。
- 需要确认：项目负责人或协作方反馈。
- 不能确认：客服工单模板是否降低重复定位成本。
- 需要确认：客服团队使用记录、项目复盘、主管反馈。

## 9. Governance Record

治理记录是轻量 trace，不是合规审计系统。它要增强可信度和边界感。

位置：右侧侧栏的第二个视角，或材料包下方的折叠记录区。

展示内容：

- AI 输出时间：例如 2026-04-26 10:40，演示时间。
- 输入材料范围：KPI 摘要、绩效评级、项目复盘、知识库记录、mentor 记录、跨团队协作评价、员工补充。
- 已验证材料：mentor 记录、知识库记录、项目复盘。
- 待确认材料：跨团队协作评价、员工补充说明。
- 员工自述处理：已结构化为待确认线索，未进入已验证证据。
- 人工复核边界：AI 不确认未进入内部材料的事实，不判断是否达到晋升标准。

治理记录文案要避免“审计通过”“合规确认”“系统已确认”等真实流程闭环表达。

## 10. Employee Supplement Loop

员工补充入口仍只在候选人 B 深度案例中出现。

布局：

- 左侧：员工原始补充说明。
- 右侧：AI 结构化为待确认线索。
- 下方：缺失证据提示和治理状态。

示例原始补充：

> 我还参与了客服工单归因模板的整理，和体验团队一起把高频问题拆成了 6 类。后来客服同学说定位重复问题快了很多，但这个没有写进项目复盘。

结构化结果：

- 贡献维度：用户价值、知识沉淀。
- 贡献线索：参与整理客服工单归因模板，可能帮助客服更快定位重复问题。
- 当前状态：待确认。
- 不能确认的事项：实际效率提升幅度、使用范围、是否进入正式流程。
- 建议补充来源：体验团队反馈、项目复盘记录、客服团队使用记录。
- 治理状态：员工自述，未验证，不直接进入评审事实。

按钮文案：

- 推荐：生成待确认线索。
- 不要用：提交申诉、申请复议、采纳为证据、确认通过。

## 11. Candidate A And C Summary Cases

A/C 必须保留在升级后的 case file 体系中，证明产品不是单向加分工具。

### 11.1 Candidate A Summary

候选人 A 是“维持推荐”摘要案例。

展示内容：

- 原 AI 建议：推荐晋升。
- 校准状态：维持推荐。
- 材料覆盖：显性绩效充分，隐性贡献未发现明显遗漏。
- 材料包：摘要材料。
- 下一步：维持原讨论重点。

它不需要员工补充闭环，也不需要完整材料包三视角。可以展示轻量材料包摘要，说明“未提示新增复核风险”。

### 11.2 Candidate C Summary

候选人 C 是“补充评审”摘要案例。

展示内容：

- 原 AI 建议：推荐晋升。
- 校准状态：补充评审。
- 材料覆盖：显性业绩强，协作反馈、知识沉淀和带教稳定性存在未确认点。
- 建议追问：协作团队、团队负责人、mentor 对象。
- 治理记录：AI 不能确认返工原因和知识沉淀复用范围。

文案：

> 补充评审用于完善材料结构，不是否定原推荐，也不是负面裁定。

## 12. Interaction Model

默认进入：

1. 打开晋升评估中心 AI 校准模块。
2. 显示当前评审批次概览。
3. case file 队列中候选人 B 默认选中。
4. 中间显示候选人 B 的 AI 校准工作台。
5. 右侧默认显示 AI 评审材料包的评审委员版。

核心交互：

- 切换 case file：A/B/C 切换候选人内容和材料包。
- 查看校准链路：点击链路节点可滚动到对应模块，或仅作为静态导航提示。
- 证据筛选：全部、已验证、待确认、证据不足。
- 材料包视角切换：评审委员版、员工解释版、人工复核版。
- 治理记录切换：从材料包切到治理记录。
- 员工补充生成：点击“生成待确认线索”后显示结构化结果，状态保持待确认。

不要加入：

- 提交审批。
- 通过/驳回。
- 申诉。
- 排名排序。
- 自动改分。
- 真实保存状态。

## 13. Key States

默认状态：候选人 B 已完成 AI 校准分析，材料包已生成，治理记录可查看。

校准分析加载状态：若需要展示，使用短暂骨架屏，文案为“正在整理材料覆盖与贡献线索”。不要展示模型日志。

材料包生成状态：可用轻量状态“材料包已生成”，不要表现为正式归档或审批。

员工补充生成状态：显示“已整理为待确认线索”，不能显示“已采纳”。

空状态：某类证据没有材料时显示“当前材料中未发现可确认线索”，不要显示“无贡献”。

证据不足状态：说明缺少什么材料，例如“缺少项目复盘来源，暂不能确认影响范围”。

错误状态：文案为“材料整理未完成，请稍后重试或查看原评估结果”。不要暗示评审流程中断。

## 14. Visual Quality Requirements

这个 Demo 需要在竞品中脱颖而出，关键是专业密度和证据链设计，而不是装饰。

必须做到：

- 有明确系统外壳，但不膨胀成完整 HR 平台。
- 批次、队列、工作台、材料包、治理记录五层结构清楚。
- case file 看起来像真实评审工作项，不像普通卡片。
- 证据区像 workpaper，有来源、状态、缺口、动作、材料包归属。
- 材料包像评审会可直接使用的材料，不像 AI 作文。
- 治理记录像可信 trace，不像合规审计产品。
- 所有 AI 边界提示显眼但不过度警告。

避免：

- 一页到底的卡片堆叠。
- HR dashboard 大屏化。
- AI 聊天框化。
- 自动评分器化。
- 申诉系统化。
- 装饰性 AI 风格，例如渐变光效、发光芯片、魔法棒叙事。
- 只写漂亮总结，没有可验证证据和下一步动作。

禁止或避免：

风险提示句：

员工补充句：

> 员工补充说明会被整理为待确认线索，需结合主管、协作方或项目材料进一步确认。

## 16. Demo Content Boundaries

可以使用 mock data 和 mock AI output，但必须显得像真实企业材料。

Mock data 应包含：

- 当前晋升周期和评审批次。
- 3 个候选人 case file。
- 每个 case file 的原 AI 建议、校准状态、证据覆盖、未确认事项和下一步动作。
- 候选人 B 的完整证据 workpaper。
- 候选人 B 的 AI 评审材料包三视角。
- 候选人 B 的治理记录。
- 候选人 B 的员工解释和员工补充结构化结果。
- 候选人 B/C 的建议追问问题。

不要加入：

- 真实员工隐私数据。
- 私人聊天内容。
- 真实数据库。
- 登录权限。
- 审批流。
- 申诉系统。
- 员工排名。
- 模型训练。
- 真实审计或合规留痕。

## 17. Acceptance Checklist For The UI

- 用户能感知这是晋升评估中心中的 AI 校准模块，而不是孤立单页。
- 用户先看到评审批次和 case file 队列，再进入候选人 AI 校准工作台。
- A/B/C 三种 case file 场景清楚可辨：维持推荐、低估风险、补充评审。
- 候选人 B 默认选中，并展示完整深度闭环。
- 候选人 B 工作台包含：原 AI 评估、材料覆盖盲区、原评估与校准关注点对比、隐性贡献证据、低估风险提示、AI 材料包、员工解释、员工补充、待确认线索、人工复核边界。
- 证据 workpaper 中每条证据都有来源、状态、缺失证据、下一步动作和材料包归属。
- 被低估风险被表达为复核提示，不是 AI 改分或晋升结论。
- AI 评审材料包区分评审委员版、员工解释版和人工复核版。
- 治理记录展示材料来源、AI 输出时间、员工自述状态和人工确认边界。
- 员工解释中性、可理解、不过度承诺。
- 员工补充默认进入待确认，不直接成为事实。
- 候选人 A/C 保持摘要案例，且仍在 case file 体系中表达清楚。
- 页面没有真实登录、审批、申诉、排名、数据库、模型训练或完整 HR 平台入口。
- 页面文案不出现“AI 修正结果”“AI 改分”“公平裁判”“自动晋升”等误导表达。
